日期：2026-06-24

## 结论

当前项目已经不再是“功能堆叠的小说工具”，而是基本形成了一个更清楚的产品核：

```text
粘贴第一章 -> 找出最大流失点 -> 给出正文证据 -> 生成改稿 Prompt -> 改完复诊
```

这个核是对的，应该继续强化。

但是，已实现能力过多，用户第一次进来仍可能误以为这是“复杂写作工作台”。整书拆解、关系图谱、研究库、导出包是有价值的进阶能力，但不能抢第一层心智。下一阶段不应该继续横向加功能，而应该把核心诊断闭环做得更可信、更短、更可传播。

一句话产品纪律：

```text
凡是不能帮助用户更快回答“我的第一章为什么没人追、下一版先改哪里”的功能，都不能出现在第一层。
```

## 评审团意见

### 产品评审

产品核心成立，但卖点必须从“功能很多”切到“诊断可信”。

用户不是缺一个能写小说的 AI，而是缺一个能告诉他“为什么写出来没人追”的外部评审。当前 30 秒小说诊断、正文证据、改稿 Prompt、复诊对比，正好能避开一键写作工具的主战场。

最大风险是：进阶功能太多，会稀释“审稿台”的心智。整书拆解和关系图谱很适合做第二卖点，但不能变成第一卖点。用户应该先被一句话打中，再看到进阶能力。

产品建议：

- 第一层只卖“第一章诊断 + 改稿 Prompt + 复诊”。
- 第二层卖“深度质检：成熟样本 Rubric + 证据链评分”。
- 第三层卖“AI 拆书：关系图谱、时间线、世界书、导出资产”。
- 研究库和多书对比暂时不要主推，等样本沉淀和案例足够后再推。

### UI 设计评审

当前首页比之前收敛很多，左侧主导航只保留急诊室、AI 设置、深度质检，是正确方向。

仍然存在三个 UI 问题：

- 首屏还有“状态”“当前模型”“三步卡片”“诊断输入”“下一步卡片”，信息层级偏多。
- AI 设置作为主导航项必要，但对普通作者来说不应先被注意到；可以改成状态条中的“切换模型”入口。
- 深度质检页面已经把复杂能力折叠，但顶部步骤仍像流程说明，缺少“为什么现在不建议你做高级质检”的明确判断。

UI 建议：

- 首页首屏保留一个主任务：粘贴第一章并生成诊断。
- 状态和模型卡压缩为一行小状态，不再占据首屏主视觉。
- 三步卡片保留，但放到输入框下方或结果后，避免抢任务入口。
- 深度质检页顶部明确写：“急诊跑通后再用，不是新手第一步”。
- 整书拆解页入口继续隐藏，只从结果页或 README 进阶路径进入。

### 架构评审

后端能力已经明显超过 MVP：快速诊断、参考画像、Rubric、评分、整书 Map-Reduce、上传预处理、任务恢复、证据搜索、导出、研究库、模型供应商测试都已经存在。

这说明技术上已经可行，但架构风险开始从“能不能做”转向“结果是否可信、状态是否可恢复、契约是否稳定”。

架构建议：

- 把 `quick-review` 作为稳定核心契约，先冻结字段语义。
- 为诊断结果增加证据锚点：原文片段、位置、问题类型、置信度、建议动作。
- 复诊不应只是重新跑一次，而应有明确的版本对比结构。
- 整书图谱修正目前是前端本地修正，后续如果继续主推图谱，必须持久化。
- 研究库 Q&A 不应先扩展，应先保证整书拆解结果的可追溯质量。

### 开发评审

项目工程化基础不错：monorepo、Web/API/core 分层、类型契约、测试、lint、start-local、CI、Docker、One CLI 都已覆盖。

当前开发短板不是缺框架，而是缺产品质量评测：

- 缺少固定小说样本的 golden tests。
- 缺少诊断结果的结构化质量评分。
- 缺少从输入章节到诊断报告再到复诊的端到端用例。
- 缺少模型失败、JSON 修复失败、证据缺失时的用户可见降级策略测试。

开发建议：

- 建立 `fixtures/novel-diagnosis` 样本集，每个样本标注预期最大问题。
- quick-review、score、book-map、graph-export 都加 golden snapshot。
- 增加 Playwright e2e：首次进入、填示例章节、生成诊断、复制 Prompt、复诊入口。
- 把 demo/mock 结果做成营销和测试共用资产。

## 必须补全的核心能力

### 1. 诊断可信度

当前最该补的是“为什么 AI 的判断可信”。

需要新增结果结构：

```text
问题类型
-> 原文证据
-> 读者可能反应
-> 严重程度
-> 修改动作
-> 复诊检查点
```

验收标准：

- 每个主问题至少绑定 1-3 段原文证据。
- 每条建议都能解释“如果不改，读者会怎么流失”。
- 低置信度结论必须标记“需要人工判断”，不能装作确定。

### 2. 复诊闭环

复诊是产品差异点，但现在还不够像一个闭环产品。

需要补：

- 保存改前/改后版本。
- 对比最大流失点是否变化。
- 对比 quickScore 是否提升。
- 对比改稿 Prompt 是否完成。
- 输出“下一版还剩一个最重要问题”。

验收标准：

- 用户能看到“上一版问题 -> 这一版变化 -> 下一版动作”。
- 复诊结果不是简单再跑一次报告，而是明确判断这次修改是否有效。

### 3. 示例章节和示例报告

陌生用户最怕空白输入框。当前“填入示例章节”已经有按钮，但还需要把示例变成产品演示资产。

需要补：

- README 中提供一个“示例输入 -> 示例诊断”的短截图或链接。
- 应用内提供 2-3 个题材示例：玄幻、都市、言情。
- 示例报告可用于测试、SEO 页面和社交传播。

验收标准：

- 新用户不需要自带作品，也能在 30 秒内看到一份完整报告。
- 示例报告能体现正文证据、改稿 Prompt 和复诊检查点。

### 4. 关系图谱准确率与修正持久化

关系图谱是第二卖点，但必须建立在可信抽取上。

需要补：

- 节点、边、时间线都绑定章节出处和证据片段。
- 合并节点、忽略节点、修改关系标签要持久化。
- 导出图谱时保留 correction 记录和证据来源。
- 对低证据边自动进入复核队列。

验收标准：

- 用户能解释每条边从哪里来。
- 人工修正后刷新页面不丢。
- 导出的图谱不是漂亮图，而是可追溯学习资产。

### 5. 首屏减负

当前首屏已经可用，但还可以继续砍。

需要调整：

- 状态卡和模型卡合并为一行。
- 三步卡片移到输入区后或结果后。
- 首页只保留一个主 CTA：生成改稿方案。
- AI 设置不作为普通用户第一注意对象。

验收标准：

- 首屏 5 秒内，用户只需要理解一件事：贴第一章。
- 不阅读说明也能完成第一次诊断。

## 三期迭代计划

### 第一期：把“小说诊断台”打穿

周期：1-2 周。

目标：让用户第一次使用就觉得“它真的指出了我这章的问题”。

范围：

- quick-review 结果增加证据锚点、问题类型、严重程度、置信度。
- 首页继续减负，只保留第一章输入、生成按钮、示例入口、模型状态。
- 复诊对比做成独立结果区。
- 示例章节和示例报告固化为 fixtures。
- README 增加“示例诊断报告”入口。

不做：

- 不新增更多导出格式。
- 不扩展研究库问答。
- 不新增账号系统。
- 不把整书拆解放回主导航。

### 第二期：把“深度质检”变成可信证据链

周期：2-4 周。

目标：让严肃作者愿意把它用于改稿决策，而不是只看热闹。

范围：

- 参考样本画像改成“目标读者期待模型”。
- Rubric 输出解释“这个标准从哪里来”。
- 评分报告统一为“问题 -> 证据 -> 影响 -> 改法”。
- 建立 10-20 个题材样本的 golden tests。
- 增加端到端测试：示例章节 -> 急诊 -> 深度质检 -> 复诊。

不做：

- 不把平台算法预测包装成确定结论。
- 不让用户先填复杂数据。
- 不做通用 AI 写作助手。

### 第三期：把“AI 拆书图谱”做成进阶卖点

周期：4-8 周。

目标：把整书拆解从“功能很酷”变成“新手学习成熟作品的工具”。

范围：

- 整书节点/关系/时间线全部证据回链。
- 图谱修正持久化。
- 低证据关系复核队列。
- 一键导出学习报告、关系图图片、世界书、不要照搬清单。
- 做一个可公开展示的 demo 书籍拆解样例。

不做：

- 不把拆书包装成仿写或洗稿工具。
- 不优先扩展多书聊天。
- 不把所有导出格式都放在第一层。

## 砍功能原则

后续每个功能都要过这四个问题：

1. 它是否能更快回答“为什么没人追”？
2. 它是否能给出可验证的正文证据？
3. 它是否能转化成下一版改稿动作？
4. 它是否会增加第一次使用的理解成本？

只要第 4 条为“是”，且前 3 条不能明确成立，就应该下沉、暂停或删除。

## 下一步执行清单

P0：

- 首页首屏继续减负。
- quick-review 增加证据锚点和置信度。
- 复诊版本对比产品化。
- 示例章节和示例报告固化。

P1：

- 深度质检统一证据链结构。
- 建立 golden fixtures 和 e2e。
- AI 设置下沉但保留可达。
- 导出中心收缩默认格式。

P2：

- 图谱修正持久化。
- 研究库按样本数量条件展示。
- 多书对比和研究问答转为“样本足够后”的高级能力。
- 公开 demo 拆书案例。

## 风险边界

- 当前不能承诺“预测平台流量”，只能诊断文本是否浪费点击。
- 当前不能承诺“评分绝对准确”，必须以证据链和可复诊为可信基础。
- AI 拆书必须坚持“学结构，不搬内容”，导出内容要持续保留避险提示。
- 本地单人使用仍是当前默认场景，公开 SaaS 化需要另做账号、隔离、计费、隐私和版权治理。
